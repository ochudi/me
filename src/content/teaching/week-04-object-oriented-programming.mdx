## The problem classes solve

Keep a student's name in one variable and their score in another and
nothing ties them together. Pass them to three functions and you have
three chances to mix up the order. A class makes the grouping explicit
and gives the data its own behaviour.

```python
class Student:
    def __init__(self, name: str, score: int) -> None:
        self.name = name
        self.score = score

    def passed(self) -> bool:
        return self.score >= 50


ada = Student("Ada", 72)
print(ada.passed())
```

The object carries its own rules. Nobody computes pass marks by hand at
the call site, so the rule lives in exactly one place.

## The same idea in Rust

Rust separates the data definition from the behaviour block, but the
idea is identical: state and the functions that govern it travel
together.

```rust
struct Student {
    name: String,
    score: u32,
}

impl Student {
    fn new(name: &str, score: u32) -> Self {
        Student { name: name.to_string(), score }
    }

    fn passed(&self) -> bool {
        self.score >= 50
    }
}

fn main() {
    let ada = Student::new("Ada", 72);
    println!("{}", ada.passed());
}
```

Notice what Rust adds: the compiler checks who may touch the fields and
when. Python trusts you. Neither choice is free.

## Encapsulation

Code outside the class should go through methods, not reach into fields.
In Python this is a convention (a leading underscore); in Rust it is
enforced (fields are private to the module by default). The habit to
build this week: ask what an object should refuse to let you do.
